# 코드 스타일

## 목적

이 문서는 `ComfyUI_AOG`에서 코드를 수정하거나 새 노드를 추가할 때 따를 기본 스타일 규칙을 정리한다.
가독성, 유지보수성, 그리고 워크플로우 중심 디버깅 편의성을 우선한다.

## 기본 원칙

### 1. 노드는 짧고 분리된 책임을 가진다

한 노드는 한 단계의 역할만 담당한다.
예를 들면 아래처럼 책임을 나눈다.

- 비디오 로드/정규화
- 비디오 특징 추출
- QwenVL 분석
- 프롬프트 작성
- 가사 작성
- 음악 계획 생성
- ACE-Step 합성
- MMAudio SFX 합성
- 최종 오디오 믹스
- 저장/프리뷰

### 2. 오케스트레이션과 기능 구현을 분리한다

실제 기능 노드는 독립적으로 재사용 가능해야 한다.
토글, 분기, 실행 건너뛰기 같은 제어 로직은 가능한 한 상위 orchestration 레이어나 gate 노드에서 처리한다.

### 3. 숨은 규칙보다 명시적인 계약을 우선한다

입력/출력 타입, 기본값, summary JSON 구조, 저장 경로 정책은 코드와 문서에서 명확하게 드러나야 한다.

## 주석 규칙

### 1. 주석은 필요한 곳에만 쓴다

주석은 코드가 이미 말해주는 내용을 반복하지 않는다.
아래 같은 경우에만 짧고 직접적으로 남긴다.

- 왜 이런 분기나 우회 처리가 필요한지 설명해야 할 때
- 외부 노드나 모델의 제약 때문에 특별한 처리를 할 때
- 저장/프리뷰/게이트처럼 디버깅 포인트가 되는 흐름일 때
- 메모리 정리, fallback, skip sentinel 같은 비직관적 동작이 있을 때

### 2. 주석은 한국어로 쓴다

프로젝트 내부 주석은 한국어를 기본으로 한다.
필요할 때만 모델명, API 이름, 타입명 같은 고유 명사는 원문을 유지한다.

### 3. 주석은 짧고 단정하게 쓴다

장황한 설명보다 바로 이해되는 한두 줄이 좋다.

좋은 예:

```python
# SFX가 꺼져 있어도 저장 체인은 유지해야 하므로 무음 오디오로 통과시킨다.
```

나쁜 예:

```python
# 여기서는 sfx가 false일 수도 있는데 그 경우 downstream node가 실패하면 안 되므로
# 저장이 계속 되게 하기 위해 silent audio를 만들고 이를 리턴한다.
```

## 독스트링 규칙

### 1. 공개 함수와 메서드에는 독스트링을 작성한다

아래 대상에는 독스트링을 작성한다.

- 모듈 레벨 공개 함수
- 노드 클래스
- 노드의 `FUNCTION` 메서드
- 재사용되는 핵심 헬퍼 함수

### 2. 독스트링은 한국어로 쓴다

기본 설명은 한국어로 쓴다.
타입명이나 외부 라이브러리 이름은 필요 시 원문을 유지한다.

### 3. 기본 형식은 아래를 따른다

독스트링은 아래 구조를 기본으로 한다.

```python
"""함수 동작 설명.

Args:
    arg_name: 인자 설명.

Returns:
    반환값 설명.
"""
```

### 4. `Args`와 `Returns`는 필요할 때만 쓴다

- 인자가 있으면 `Args:`를 쓴다.
- 반환값이 있으면 `Returns:`를 쓴다.
- 인자나 반환값이 없으면 해당 섹션은 생략한다.

즉 모든 독스트링에 무조건 `Args`와 `Returns`를 넣는 것이 아니라,
실제 존재하는 경우에만 넣는다.

### 5. 한 줄 독스트링으로 끝내도 되는 경우

아주 단순한 클래스나 함수는 아래처럼 짧게 끝내도 된다.

```python
"""AOG 오디오 요약 JSON을 저장한다."""
```

단, 공개 노드의 핵심 메서드처럼 입력/출력이 중요한 경우에는
반드시 `Args:`와 `Returns:`를 포함한 형식을 사용한다.

## 독스트링 예시

### 예시 1. 인자와 반환값이 모두 있는 경우

```python
def compose(video_features, prompt_text):
    """비디오 특징과 프롬프트를 사용해 ACE-Step 입력 조건을 만든다.

    Args:
        video_features: 비디오 분석 결과 payload.
        prompt_text: 최종 프롬프트 문자열.

    Returns:
        ACE-Step 조건 생성에 사용할 요약 dict.
    """
```

### 예시 2. 인자만 있고 반환값이 없는 경우

```python
def log_summary(summary_json):
    """요약 JSON을 디버그 로그에 남긴다.

    Args:
        summary_json: 저장 또는 출력할 요약 문자열.
    """
```

### 예시 3. 반환값만 있고 인자가 없는 경우

```python
def build_default_prompt():
    """기본 SFX 프롬프트를 만든다.

    Returns:
        기본 SFX 프롬프트 문자열.
    """
```

## 노드 구현 규칙

### 1. 가능한 한 순수한 입력/출력을 유지한다

노드는 입력으로 받은 값에서 예측 가능한 출력을 만들어야 한다.
숨은 전역 상태 의존은 최소화한다.

### 2. summary JSON은 디버깅 가능한 정보를 담아야 한다

각 단계의 summary JSON에는 최소한 아래 중 필요한 항목이 들어가야 한다.

- 실제로 사용된 프롬프트/가사
- 자동 생성 여부
- LLM provider 정보
- fallback 또는 degraded 경고
- 출력 길이, 샘플레이트, 적용된 gain
- 토글에 의해 skip 되었는지 여부

### 3. 저장/프리뷰 경로는 실제 ComfyUI 출력 계약을 존중한다

`OUTPUT_NODE`를 쓰는 노드는 ComfyUI와 외부 커스텀 노드가 기대하는 반환 형식을 깨지 않도록 주의한다.
특히 VHS 계열 노드는 `ui` payload와 `result` payload를 임의로 단순화하지 않는다.

## 워크플로우 JSON 규칙

### 1. 실제 노드 시그니처와 widget 순서를 반드시 맞춘다

`widgets_values` 순서가 `node.py`의 입력 정의와 다르면 GUI 복원과 실행이 깨진다.
워크플로우를 수정한 뒤에는 반드시 실제 노드 정의와 순서를 다시 점검한다.

### 2. 기본값은 “업로드 후 바로 실행 가능”을 목표로 한다

예시 워크플로우는 영상만 업로드하면 바로 큐에 넣을 수 있어야 한다.
불필요한 수동 입력은 기본 워크플로우에서 제거한다.

### 3. 저장 경로와 preview 노드는 항상 검증한다

워크플로우 변경 후에는 아래를 실제로 확인한다.

- 파일이 `ComfyUI/output` 아래에 저장되는지
- preview가 UI에 표시되는지
- summary JSON이 함께 저장되는지

## 검증 규칙

코드 수정 후에는 가능하면 실제 ComfyUI 런타임으로 아래를 확인한다.

- full 옵션
- ACE-Step only
- MMAudio SFX only
- prompt/lyrics UI 표시
- summary JSON 저장
- preview 표시

문법 검사만으로 끝내지 않는다.
